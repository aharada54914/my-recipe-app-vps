# Kitchen App PWA 仕様書（0から再設計版）

最終更新: 2026-03-07（v2.0.0 / Phase 5-6 反映）

本書は要件定義書をもとに、実装観点（How）を定義する。

---

## 1. システム構成

## 1.1 全体アーキテクチャ

- フロントエンド: SPA + PWA（Service Worker）
- データ保存: IndexedDB（主）、LocalStorage（軽量設定）
- 外部連携:
  - Google OAuth
  - Google Drive API（バックアップ）
  - Google Calendar API（献立/買い物予定）
  - Gemini API（AI補助）

## 1.2 設計方針

- **Offline-First**: 読み取り中心機能はネット断でも動作
- **Local-First Data**: ユーザー操作はまずローカルへ反映
- **Progressive Enhancement**: 外部連携は任意機能として段階提供

8. 設定 `/settings/:tab`

## 2.2 主要画面要件

### Home. ホーム画面（v2.0.0 全面刷新）

- 一次導線は `レシピ検索` と `AI に相談`
- `今週の献立` はサマリーカード + compact timeline で表示
- **今日食べたい料理**（2×2タイル）: 気象庁API取得の当日天気 × 旬スコア × 個人化で評価した上位4件を表示。天気API失敗時は旬フィルタのみにフォールバック。
- **在庫でつくれるレシピ**（2×2タイル）: 在庫一致率上位4件（在庫0件時は非表示）
- **旬のおすすめ**（2×2タイル）: 月別24種プールから日付シードで10件選抜し上位4件
- カテゴリ画像タイル（2×4）からの検索遷移
- ヘッダー: 検索 / 複数レシピスケジュール / 在庫管理 / 設定またはアカウント
- 下部ナビ: `ホーム / 献立 / Gemini / お気に入り / 履歴`

### A. 検索画面

- 入力: キーワード、カテゴリ、機器、時短、旬
- 表示: レシピカード（在庫一致率、所要時間、機器タグ）
- 補助導線: 最近の検索、最近見たレシピ
- 操作: 詳細表示、お気に入り登録、献立追加

### B. 在庫画面

- 入力: 食材検索、数量
- 表示: カテゴリ別/50音一覧
- 操作: `+1 / -1 / 数量入力 / 削除`、プリセット一括登録、QR共有インポート

### C. 週間献立画面

- 入力: 期間、人数、ロック状態
- 表示: summary、注目日、残り日程、買い物リスト
- 操作: 自動生成、差し替え、再生成、カレンダー登録、共有

### D. AI画面

- 入力: URL、画像、テキスト指示
- 表示: 解析結果（材料/手順）編集フォーム
- 操作: 保存前編集、再解析、チャット連携

## 4. 機能仕様

## 4.0 ホームレコメンドエンジン（v2.0.0）

天気快適スコア、個人化スコア、旬スコアを合成してレシピをランキングする。

```
todayScore = 0.75 × unifiedWeatherScore + 0.25 × seasonalScore
unifiedWeatherScore = weather vector score + T_opt personalization score を統合した指標
```

- 天気データ: 気象庁API `/bosai/forecast/data/forecast/130000.json`（東京エリア）
- フォールバック: API失敗時は月別平均気温による合成7日間予報を使用
- 実装詳細: `docs/ALGORITHMS.md` §12〜13 を参照

## 4.1 検索エンジン

- インデックス: タイトル、材料、タグ、機器、カテゴリ
- マッチ方式: 前方一致 + あいまい一致（重み付き）
- スコア補正:
  - 在庫一致率（加点）
  - 好みシグナル（加点）
  - ヘルシオデリ等の優先度調整（減点/下位化）

## 4.2 在庫一致率計算

- 材料の正規化（全角半角・大小・別名）
- レシピ必須材料に対し、在庫有りの割合を算出
- 表示は%（整数丸め）

## 4.3 献立自動生成

- 入力: 期間、避けたいカテゴリ、人数
- 生成手順:
  1. 主菜候補抽出（在庫・時短・重複抑制）
  2. 日別主菜配置（連続カテゴリ回避）
  3. 副菜/スープ補完（主菜との相性ルール）
- ロック済み日は不変、非ロック日のみ再計算

## 4.4 買い物リスト生成

- 献立材料を日次で集約
- 在庫差分を引いて不足分のみ抽出
- 調味料は初期非表示（トグルで表示）

## 4.5 AI取り込み

- URL解析: Web本文抽出 -> Geminiで構造化JSON化
- 画像解析: OCR/視覚解析 -> 食材候補列挙
- 保存条件:
  - 必須項目（タイトル、材料1件以上、手順1件以上）
  - ユーザーが最終確認して保存操作

## 4.6 バックアップ/復元

- 形式: `kitchen-app-backup-YYYYMMDD.json`
- 内容: レシピ、在庫、献立、設定、履歴
- 復元:
  - ドライラン（件数表示）
  - 本適用（確認必須）

## 5. PWA仕様

## 5.1 マニフェスト

- `name`, `short_name`, `icons`, `start_url`, `display=standalone`
- iOS向け `apple-touch-icon` を提供

## 5.2 Service Worker

- 初回: シェル資産をプリキャッシュ
- 実行時キャッシュ戦略:
  - 静的アセット: Cache First
  - API/外部通信: Network First + フォールバック
  - 画像: Stale While Revalidate

## 5.3 通知

- 種別: 調理開始、献立生成完了、買い物確認
- スケジュール: ローカル通知を優先
- 通知許可がない場合は代替UI（バッジ/アラート）

---

## 6. 同期・競合解決

- 同期単位: ドメイン（在庫/献立/設定）ごと
- 競合ポリシー:
  - 通常: `last-write-wins`
  - 重要データ（献立）: 差分表示してユーザー選択
- 失敗時: リトライキューへ退避し、復帰後再送

---

## 7. セキュリティ仕様

- OAuthトークンは安全に保存し、平文ログを禁止
- APIキーは環境変数またはユーザー設定から投入
- 共有URLは期限/改ざん検知トークンを付与（推奨）
- 依存ライブラリ脆弱性チェックをCIで実施

---

## 8. 品質保証

## 8.1 テストレベル

- 単体テスト: スコア計算、在庫一致率、献立生成ルール
- 結合テスト: Drive復元、Calendar登録、AI保存フロー
- E2Eテスト: 検索→献立→買い物→共有の主導線

## 8.2 受け入れテスト（例）

- オフラインで検索・お気に入り閲覧可能
- 献立再生成でロック日が保持される
- 復元時のドライラン件数が実件数と一致
