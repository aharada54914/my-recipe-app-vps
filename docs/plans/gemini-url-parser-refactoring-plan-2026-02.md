# Gemini URL解析機能 改善リファクタリング計画（2026-02）

## 背景
- 一部サイト（例: bazurecipe.com）のURL取得が失敗し、GeminiのURL解析フロー全体が停止する。
- 現状は単一の取得経路（直接fetch）依存で、失敗時の代替経路がない。
- 取得・抽出・解析の責務が単一ハンドラ内に集中し、障害切り分けが難しい。

## 目標
1. URL取得成功率の改善（特にWAF/CDN配下サイト）。
2. 失敗時の原因可視化（どの取得経路で失敗したか）。
3. API層の責務分離による保守性向上。

## 非目標
- 全レシピサイトへの対応拡大。
- Geminiプロンプト設計の大幅変更。

## 現状課題（要約）
1. **単一障害点**: 直接fetch失敗で即エラー。
2. **フォールバック欠如**: HTML取得不能時の代替手段なし。
3. **観測性不足**: フロント側で失敗要因が追跡しづらい。
4. **関数肥大化**: 取得・抽出・レスポンス整形が密結合。

## リファクタリング方針
### Phase 1（即効）
- URL取得を**戦略パターン化**し、
  - Strategy A: 直接fetch
  - Strategy B: `r.jina.ai` プロキシ経由テキスト取得
  の順で試行。
- 成功時は `fetchStrategy` をレスポンスに含める。
- 失敗した試行理由を `warnings` として返却する。

### Phase 2（安定化）
- `api/recipe-extract.js` を以下へ分割：
  - `fetchers/directFetcher`
  - `fetchers/proxyFetcher`
  - `parsers/jsonLdParser`
  - `parsers/metaParser`
  - `normalizers/textNormalizer`
- 単体テスト追加（取得戦略選択、エラー集約、JSON-LD抽出）。

### Phase 3（品質向上）
- 取得失敗の統計ログ（ドメイン別成功率/失敗理由）を収集。
- 失敗時UX改善（ユーザー向け具体メッセージ + テキスト貼り付け導線）。
- 対応サイトごとの微調整ヘッダ（必要時）。

## 実装タスク
1. 取得戦略抽象化（`extractRecipeSource`導入）。
2. 直接fetchヘッダ強化（Accept-Language/Referer）。
3. フォールバック経路（jina-ai）実装。
4. エラー連結メッセージ実装（複数経路失敗時）。
5. フロント側で `warnings` をログ可視化。
6. 将来分割を前提に関数境界整理。

## 受け入れ基準
- bazurecipe URLが少なくとも片方の戦略で取得成功する。
- 失敗時に「どの経路で何が失敗したか」がエラーメッセージ/ログで追える。
- 既存の対応サイトで回帰（URL解析不能）が発生しない。

## ロールアウト
1. まず本ブランチでPhase 1を導入し、既存動作を検証。
2. QA後にPhase 2を別PRで実施。
3. 運用データを見てPhase 3着手。
