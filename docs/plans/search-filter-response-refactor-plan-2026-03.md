# 検索入力・カテゴリ切替の体感遅延を減らすための改善案とリファクタリング計画（2026-03 / revised）

## 0. 目的（今回の見直し方針）

ユーザー要望「**アニメーションはリッチなのが好き**」を反映し、

- 単なる高速化ではなく、**気持ちよい反応体験（Motion UX）** を設計する
- 遅延そのものを減らす技術改善と、待ちを短く感じさせる心理設計を両立する
- 検索・カテゴリ切替で「押した瞬間に反応した」と感じる一貫した体験を作る

---

## 1. 現状課題（再整理）

### 1-1. 実時間の課題

- 検索: `useDebounce(300)` と重いスコア計算により結果反映が後ろ倒し
- カテゴリ: フィルタ変更→DB取得→再評価→描画で待ちが連鎖
- 表示: 仮想リスト更新時に「一瞬止まった」ような知覚が発生

### 1-2. 体感の課題

- タップ/入力直後の視覚フィードバックが弱い
- 処理中の状態が不明瞭（無反応に見える）
- 切替時にレイアウトが急変し、認知コストが高い

---

## 2. 改善案（技術 + リッチアニメーション + 心理設計）

## A. 検索欄の応答性改善（即時反映 + 段階検索）

1. **入力ステートを二層化**
   - `inputValue`: 即時表示専用（1フレーム以内）
   - `committedQuery`: 検索実行専用（短デバウンス）

2. **デバウンス最適化（300ms → 120〜180ms）**
   - 通常150ms目安、低性能端末のみ180〜220msへ可変

3. **明示的確定は即時検索（デバウンス無視）**
   - Enter / 検索ボタンは `flush` 実行

4. **IME中は検索抑制**
   - `compositionstart/end` で確定前検索を停止

5. **入力中は軽量検索、停止後に高精度検索**
   - 入力中: 前方一致・token一致
   - 停止後: 同義語・読み仮名・fuzzy拡張

6. **検索結果キャッシュ（LRU）**
   - key: `query + category/device/quick/seasonal`

7. **検索スコアをWorkerオフロード（Phase 3）**
   - メインスレッドをアニメーション描画に集中

## B. カテゴリ切替の応答性改善（主菜/副菜アイコン）

8. **楽観的UIで“押した瞬間”に選択表示**
   - 結果計算前でもタグ状態を先に反映

9. **カテゴリ件数の先出し**
   - `主菜 124` のように先に件数を表示して待ちを可視化

10. **DB取得とUIフィルタを分離**
   - `useRecipeQuery`（取得）と `useRecipeFiltering`（絞り込み）を分割

11. **Dexie索引見直し + 事前取得**
   - `category`, `device`, `updatedAt` 優先
   - 主要カテゴリ結果を起動後にウォームアップ

12. **上位N件先描画（段階レンダリング）**
   - 先に8〜12件を見せ、残りは次フレームで補完

13. **`RecipeCard` 再レンダリング抑制**
   - `memo`, 安定props, 画像描画分離

## C. リッチアニメーション設計（今回の重点）

14. **モーション設計原則を固定化**
   - `Fast feedback`: 60〜120ms
   - `State transition`: 140〜220ms
   - `Content transition`: 220〜320ms
   - easing: `cubic-bezier(0.22, 1, 0.36, 1)`（easeOutQuint寄り）

15. **カテゴリアイコンに3段アニメーション**
   - (1) Press: scale 0.96 / 80ms
   - (2) Select bloom: 背景グロー + 枠線拡張 / 160ms
   - (3) Settling: 微小バウンス / 120ms

16. **リスト更新を“置換”ではなく“遷移”にする**
   - old list 80ms fade-out
   - loading skeleton shimmer 200ms
   - new list stagger-in（1行ずつ20〜30ms遅延）

17. **検索実行時の波及アニメーション**
   - 検索バー右端アイコンにスイープ（120ms）
   - 結果件数バッジをカウントアップ（0.25s）

18. **タップ反応の触覚強化（対応端末のみ）**
   - `navigator.vibrate(6)` を任意ON設定で提供

19. **遅延時専用ローディング演出**
   - 100ms超: subtle spinner
   - 300ms超: skeleton + 文言
   - 900ms超: progress flavor text（例: 「在庫も含めて最適化中」）

20. **Reduced Motion対応**
   - `prefers-reduced-motion` でduration短縮/無効化

## D. 心理的改善（待ちを短く感じさせる）

21. **状態を言語化**（検索中 / 絞り込み中 / 並び替え中）
22. **「処理の価値」を表示**（在庫・好み反映を説明）
23. **無反応と失敗の分離**（タイムアウト時に再試行ボタン）
24. **予測可能な一貫動作**（同操作で同演出・同テンポ）
25. **空状態の先回り**（最近見た / よく作る / 季節の提案）

## 3. リファクタリング計画（再構成）

### Phase 1: 「即レス感」最優先（1〜2日）

- `inputValue` / `committedQuery` 分離
- Debounce 150ms化 + Enter即時実行
- カテゴリ楽観的UI
- 100ms超でのみローディング表示
- Press/Select の最小リッチ演出実装

**KPI**
- 入力反映 p95 < 16ms
- タップ後に選択状態が見えるまで p95 < 50ms
- 体感アンケート: 「反応した感」が改善

### Phase 2: 「遅延の実体」を削る（3〜5日）

- DB取得/絞り込み/ランキングの責務分離
- キャッシュ導入、Dexie索引最適化
- 上位N件先描画 + 段階評価
- `RecipeCard`再レンダリング削減
- リストクロスフェード + stagger導入

**KPI**
- カテゴリ切替→結果表示 p95 < 220ms
- 同条件再検索 p95 < 100ms
- Long Task回数の有意減少

### Phase 3: 「高負荷でも滑らか」を実現（1週間）

- Worker検索基盤
- 事前インデックス構築
- リッチ演出の最適化（GPU-friendly transform中心）
- モーションデザイントークン化（duration/easing/shadow）

**KPI**
- 中位端末で連続操作時 55fps以上維持
- アニメーション有効時でも入力遅延悪化なし

## 4. 実装設計（具体）

### 4-1. コンポーネント責務分割

- `useRecipeQuery` : IndexedDB取得、索引活用、事前取得
- `useRecipeFiltering` : query/category/device/quick/seasonal
- `useRecipeRanking` : スコア計算、段階評価、キャッシュ
- `useInteractionFeedback` : ローディング状態、遅延段階、文言
- `useMotionPreset` : モーション設定、Reduced Motion切替

### 4-2. Motionトークン（例）

- `--motion-press: 80ms`
- `--motion-fast: 140ms`
- `--motion-base: 200ms`
- `--motion-rich: 280ms`
- `--ease-emphasis: cubic-bezier(0.22, 1, 0.36, 1)`

### 4-3. 表示戦略

- ローディングは**遅い時だけ**見せる（常時スピナー禁止）
- リスト切替はフェード + スタッガーで視線誘導
- 重要アクション（カテゴリ）にだけ強い演出を集中

## 5. 計測・検証

### 計測イベント

- `search_input_to_visual_ack_ms`
- `search_commit_to_result_ms`
- `category_tap_to_visual_ack_ms`
- `category_tap_to_result_ms`
- `animation_dropped_frames`
- `long_task_count`

### 実験

- A/B1: 150ms debounce vs 300ms debounce
- A/B2: リッチ演出ON vs 最小演出
- A/B3: skeletonあり vs なし

### 主観評価

- 「速い」ではなく「気持ちよく反応する」評価軸を追加
- 5段階 + 自由記述（“待ちが気になる瞬間”を収集）

---

## 6. 優先順位（今回の提案）

1. **即時承認（楽観表示 + Press反応）**
2. **検索・カテゴリ処理の分離と軽量化**
3. **リッチ遷移（クロスフェード + stagger + 件数演出）**
4. **Worker化と索引化で土台強化**

この順で進めると、初週で「反応が良くなった」を体感させつつ、
中長期で本当に速い実装へ育てられる。
