# レビュー結果に基づくリファクタリング計画（優先度付き）

最終更新: 2026-02

## 0. 前提

- 入力レビューの総合スコアは **72/100**。
- 最大のボトルネックは **Performance 36/100**。
- 本計画では、依頼に従い **`meta viewport` のズーム制限修正はスコープ外** とする。
- 目的は「まず体感速度を改善し、その後に品質・運用を強化する」こと。

---

## 1. 目標（次回評価までの到達ライン）

### 定量目標

1. Lighthouse Performance: **36 → 70+**
2. LCP: **10.4s → 3.0s 以下**（モバイル計測）
3. TBT: **1,070ms → 300ms 以下**
4. 初期JS（エントリ + 直下同期読み込み）: **4.6MB相当 → 1.2MB以下**
5. アクセシビリティ: **80 → 90+**（コントラスト・見出し順序を解消）
6. テスト: 主要画面のコンポーネントテスト + 主要導線のE2Eを追加

### 定性目標

- 初回起動時の「重さ」を解消し、検索・画面遷移・献立作成の応答性を向上。
- 障害時のユーザー通知を明確化（トースト、エラーバナー、再試行導線）。
- 将来の機能追加で性能退行を起こしにくい構成にする。

## 3. フェーズ別実行計画

## Phase 1（最優先）: パフォーマンス救済

### 1-1. レシピデータの「JS同梱」をやめる

**やること**
- `prebuild-recipes.mjs` で生成した成果物を、アプリ本体JSへ埋め込まず **静的JSONとして配信**。
- 初期表示に必要な最小データのみ先読みし、残りは `fetch()` で遅延ロード。
- IndexedDB（Dexie）へキャッシュし、2回目以降のロードを高速化。

**実装方針（案）**
- `public/data/recipes-index.json`（軽量インデックス）
- `public/data/recipes-*.json`（カテゴリやページ単位に分割）
- 検索時/画面遷移時に必要チャンクだけロード

**受け入れ条件**
- `dist` のエントリJSが大幅縮小（4.6MB級の解消）。
- オフライン時は既存キャッシュ + IndexedDBから読み込み可能。

### 1-2. ルートベースのコード分割

**やること**
- `React.lazy` + `Suspense` でページ単位に遅延読み込み。
- `Settings`, `History`, `Favorites`, `AskGemini` など低頻度ルートを分割。

**受け入れ条件**
- 初期レンダリングに不要なページコードがメインチャンクから外れる。
- 初回表示のTBT改善。

### 1-3. フォント最適化

**やること**
- Google Fontsの読み込みを見直し（使用ウェイト最小化、`display=swap`）。
- 可能ならセルフホスト化または `preconnect`/`preload` を導入。

**受け入れ条件**
- レンダリングブロッキングの警告減少。
- FCP短縮。

---

## Phase 2: UX・アクセシビリティ・信頼性

### 2-1. エラーハンドリングの可視化

**やること**
- API呼び出し失敗時のトースト通知を統一実装。
- 主要画面に「再試行」導線を追加。
- ルート/ページ単位に Error Boundary を導入。

**受け入れ条件**
- 失敗時に無反応状態が残らない。
- ログ（console）依存ではなくUIで状態が伝わる。

### 2-2. アクセシビリティ改善（スコープ内のみ）

**やること**
- コントラスト不足箇所の修正（ボタン、補助テキスト等）。
- 見出し階層を `h1 -> h2 -> h3...` の順序に整理。
- *注*: viewportズーム制限修正は今回スコープ外。

**受け入れ条件**
- Lighthouse Accessibility 90+ を目標。
- キーボード操作時の視認性が維持される。

### 2-3. PWAの導入体験改善

**やること**
- Manifest `lang` を `ja` に調整。
- `screenshots` を定義。
- `beforeinstallprompt` を使ったカスタム導線（任意）を追加。

**受け入れ条件**
- インストール時の情報量と導線が改善。

---

## Phase 3: 品質と運用の底上げ

### 3-1. テスト戦略の拡張

**やること**
- React Testing Libraryで主要コンポーネントの操作テストを追加。
- E2E（Playwright/Cypress）で以下の導線を自動化:
  - レシピ検索
  - 週間献立作成
  - 買い物リスト生成
  - オフライン復帰時の表示

**受け入れ条件**
- PR時に最低限の回帰が検知できる。
- 重要導線のテストがCI上で安定実行。

### 3-2. 依存関係と脆弱性対応

**やること**
- `npm audit fix` を基本に、破壊的変更は段階的に適用。
- High優先で解消し、必要に応じて代替ライブラリも検討。

**受け入れ条件**
- High脆弱性の件数を段階的に削減。
- 依存更新後もビルド・テストが通る。

## 5. リスクと対策

1. **データ分割で検索体験が劣化するリスク**
   - 対策: 軽量インデックスを先読みし、詳細は遅延取得。
2. **コード分割でローディング体験が悪化するリスク**
   - 対策: `Suspense` フォールバックを画面別に最適化。
3. **依存更新で破壊的変更が入るリスク**
   - 対策: 小分けPR + 自動テスト + ロールバック手順。

---

## 6. すぐ着手する具体タスク（次の1スプリント）

1. `prebuild-recipes.mjs` の出力を「配信用JSON + 参照インデックス」形式に変更
2. レシピ取得層を `loadRecipeIndex()` / `loadRecipeChunk()` に分離
3. `App.tsx` のルーティングを `React.lazy` 化
4. フォント読み込みの見直し（ウェイト削減 + preload/preconnect）
5. Lighthouse計測をPRテンプレに追記（before/after比較必須化）
