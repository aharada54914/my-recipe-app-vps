# 献立生成失敗インシデント: 原因究明とリファクタリング計画

## 1. 症状
- 週間献立画面の「再生成」で `献立の生成に失敗しました` ダイアログが表示される。
- エラーの導線は `WeeklyMenuPage.handleGenerate` の `catch` で捕捉される。

## 2. 技術的な原因（今回確認できた再発要因）

### 原因A: 旧ユーザー設定（legacy preferences）で `seasonalPriority` が欠損した場合の耐性不足
- `selectWeeklyMenu` は `SEASONAL_WEIGHTS[config.seasonalPriority]` を前提にスコアリングする。
- 旧データや部分更新により `seasonalPriority` が欠損すると `undefined` が入り、スコアリングの品質低下・不安定化につながる。

### 原因B: 候補不足時の失敗理由が曖昧
- `selectWeeklyMenu` は日ごとに候補が見つからない場合 `continue` し、最終結果が 7 日未満でも返しうる。
- これだと UI 側で「なぜ失敗したか」が追跡しづらく、利用者に復旧手順を提示しにくい。

### 原因C: 設定オブジェクトの正規化責務が分散
- Preferences Context で DB 値をそのまま配布すると、欠損フィールドが末端ロジックへ伝搬する。
- 生成系ロジックが各所で防御コードを持つ必要があり、保守コストが高い。

## 3. 今回の即時対策（実装済み）
1. Preferences Context で `DEFAULT_PREFERENCES` を先にマージし、欠損設定を補完。
2. `selectWeeklyMenu` 側でも `seasonalWeight` のフォールバックを追加し、二重防御化。
3. 献立選択結果が 0 件 / 7 件未満のとき、明確なエラーメッセージで例外化。
4. 候補不足時は `scoredMains` の上位から補完し、7日分を必ず構成するフォールバックを導入。

## 4. リファクタリング計画（中期）

### Phase 1: 入力境界の一本化（最優先）
- `UserPreferences` 正規化関数（例: `normalizePreferences`）を追加。
- `PreferencesProvider` とバッチ処理系で同一関数を利用し、欠損補完ルールを統一。

### Phase 2: 上位N採用戦略の明示化
- 現在のフォールバック（上位スコア補完）を仕様化し、`rankedMains` の選定ルール（重複許可条件、ロック優先順位）をドキュメント化。
- 必要に応じて週内重複率の上限を設定し、品質と完走性のバランスを調整。

### Phase 3: 監視性（observability）強化
- 生成失敗時の診断情報（候補件数、ロック件数、カテゴリ内訳）を構造化ログで収集。
- 将来的に匿名テレメトリを追加し、実機での再発率を追跡。

### Phase 4: データ品質ゲート
- レシピ投入時にカテゴリ値バリデーションを実施し、主菜/副菜ルールに合わないデータを早期検知。
- 既存 DB 向けに one-shot migration を用意して legacy 値を矯正。

## 5. 受け入れ基準（DoD）
- 欠損 `seasonalPriority` でも生成が成功すること（テストで担保）。
- 候補不足時に UI に行動可能な理由を表示できること。
- 週間献立生成の失敗率が計測可能で、失敗理由別に集計できること。
