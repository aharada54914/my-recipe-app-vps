# レシピアプリ プロジェクトレビュー

## 1. はじめに

本レビューは、QAエンジニアの視点から、プロジェクトの品質、パフォーマンス、そしてユーザー体験を評価し、改善点を指摘するものです。

全体として、仮想スクロールや検索の最適化など、パフォーマンスに関する多くのベストプラクティスが実装されており、技術的な基盤は堅固であると評価します。しかし、PWAとしての根幹機能に重大な欠陥が見られ、その他にも修正すべき点が散見されます。以下に具体的な指摘事項を詳述します。

## 2. 最重要指摘事項 (Blocker)

### 🚨 PWAの根幹機能（Service Worker）の完全な欠落

**現状:**
プロジェクト全体を調査した結果、Service Workerを登録・管理するためのコードが一切含まれていません。`vite.config.ts`に`vite-plugin-pwa`のようなPWAプラグインは導入されておらず、`main.tsx`や`index.html`にもService Workerを登録する処理が見当たりません。

**問題点:**
これは**PWA (Progressive Web App) として致命的な欠陥**です。Service Workerが存在しないため、以下のPWAの核となる機能が一切動作しません。

- **オフライン動作**: ネットワーク接続がない環境でアプリが全く起動・利用できません。
- **キャッシュによる高速化**: アセット（JS, CSS, 画像など）がキャッシュされず、毎回ネットワークからダウンロードされるため、起動時間や動作速度が著しく低下します。
- **プッシュ通知**: 将来的な機能拡張の妨げとなります。

`manifest.json`が設定されているだけではPWAとは呼べません。これは単なる「ホーム画面に追加できるウェブサイト」に過ぎず、ユーザーが期待するアプリのような体験（信頼性、速度）を提供できません。

**修正提案:**
1.  `vite-plugin-pwa`をプロジェクトに導入します。
    ```bash
    npm install vite-plugin-pwa -D
    ```
2.  `vite.config.ts`を更新し、PWAプラグインを設定します。キャッシュ戦略（`strategies`）を定義し、`manifest`をインライン化することが推奨されます。
3.  `index.html`から既存の`<link rel="manifest" ...>`を削除し、プラグインによる自動注入に任せます。
4.  Service Workerの自動更新を促すUIコンポーネントを実装します。

## 3. 重要指摘事項 (Critical)

### 🎨 アプリアイコンがデフォルトのまま

**現状:**
`public/manifest.json`および`index.html`で指定されているアイコンが、Viteのデフォルトである`/vite.svg`のままです。

**問題点:**
アプリアイコンは、ユーザーがホーム画面でアプリを識別するための最も重要な要素です。デフォルトのままでは、アプリの品質と信頼性を著しく損ないます。ユーザーに未完成品であるという印象を与えかねません。

**修正提案:**
- アプリのブランドを象徴する、適切なサイズのアイコンセット（192x192, 512x512など）を`public/icons`ディレクトリなどに配置します。
- `vite-plugin-pwa`の設定内で、これらのアイコンを`manifest`の`icons`プロパティに正しく指定してください。

### 🔄 データベース初期化処理の冗長性

**現状:**
`App.tsx`内で定義されている複数のコンポーネント（`AppShell`, `RecipeDetailPage`, `AiParsePage`など）が、それぞれ`useEffect`内で`initDb()`を呼び出しています。

**問題点:**
データベースの初期化は、アプリケーションのライフサイクルで一度だけ実行すれば十分です。各ページコンポーネントで個別に呼び出すのは冗長であり、無駄な処理です。アプリケーションのエントリーポイントで一度だけ実行するべきです。

**修正提案:**
- `App.tsx`の最上位にある`App`コンポーネント、もしくは`main.tsx`で一度だけ`initDb()`を呼び出し、初期化が完了するまでローディング画面を表示する構造に変更してください。
- `Promise`をトップレベルで`await`するか、`useState`で初期化状態を管理し、子コンポーネントに渡すのが一般的なパターンです。

## 4. 軽微な指摘事項 (Minor)

### 🧮 塩分計算ロジックの丸め処理の不一致

**現状:**
`QA.md`では「10g単位の丸め」が要求されていますが、`src/utils/recipeUtils.ts`内の`calculateSalt`関数では、塩分量(`saltG`)が`Math.round(...) / 10`によって小数点第一位に丸められています。

**問題点:**
仕様書と実装の間にわずかな乖離があります。`formatQuantityVibe`関数で最終的な表示は10g単位に丸められるため、実質的な影響は少ないかもしれませんが、計算ロジックの正確性を担保するためには、仕様と実装を一致させるべきです。

**修正提案:**
- `calculateSalt`関数内の`saltG`の計算結果を、10g単位で丸めるのではなく、要求通り小数点第一位までの精度を保つようにするか、あるいはQAドキュメントの記述を「小数点第一位までの丸め」に修正するか、どちらかに統一してください。現状では`saltG`の計算結果が`12.3`のような値になる可能性がありますが、`formatQuantityVibe`を通すと`10g`になってしまい、情報が欠落します。塩分量そのものはより正確な値を保持し、表示の際に丸めるのが適切です。

### 🧹 `react-router-dom`の非効率な利用

**現状:**
`App.tsx`の`AppShell`コンポーネント内で、`BottomNav`のタブ変更時に`navigate()`関数を呼び出してURLを制御し、さらに`activeTab`という`useState`で表示するコンポーネントを切り替えています。

**問題点:**
これは`react-router-dom`の意図された使い方ではありません。URLに応じて表示するコンテンツを決定するのがルーティングライブラリの役割です。現在の実装は、状態（`activeTab`）とURLが二重にUIを制御しようとしており、複雑でバグの温床になりやすいです。

**修正提案:**
- `Outlet`とネストされたルートを使用して、レイアウトとページコンテンツを分離してください。
- `AppShell`をレイアウトコンポーネントとし、`Header`と`BottomNav`を含めます。ページ固有のコンテンツは`<Outlet />`で描画します。
- `BottomNav`では、`useLocation`フックで現在のパスを取得し、アクティブなタブを決定します。タブのクリックでは、状態を変更するのではなく、`<Link>`または`navigate`で対応するパスに遷移させるだけにします。

## 5. 総括

本プロジェクトは、多くの高度な最適化手法を取り入れており、そのポテンシャルは非常に高いと評価します。しかし、PWAとしての基本要件が満たされていない現状は看過できません。

まずは最重要指摘事項である**Service Workerの実装**を最優先で対応してください。これが完了して初めて、PWAとしてのスタートラインに立つことができます。

厳しい指摘となりましたが、これは本プロジェクトが持つ可能性を最大限に引き出すためのものです。これらの改善点を乗り越え、ユーザーに愛される素晴らしいアプリが完成することを期待しています。

## 6. ユーザーフィードバックに基づく追加指摘事項

ユーザーからの具体的なフィードバックに基づき、以下の追加レビューを実施しました。ユーザビリティと機能性に直結する重要な課題が含まれています。

### 🐞 在庫マッチ率の計算ロジックの欠陥

**現状:**
`src/utils/recipeUtils.ts`内の`calculateMatchRate`関数が、レシピの材料のうち`category`が`'main'`のものだけを計算対象としています。しかし、多くのレシピデータでは、肉や魚などの主要な食材が`'main'`カテゴリに分類されていません。

**問題点:**
ユーザーが「鶏肉」や「豚肉」を在庫として登録しても、それらがレシピ内で`'main'`カテゴリに分類されていないため、在庫マッチ率が**常に0%と表示**されてしまいます。これは機能の根幹に関わる重大なバグであり、ユーザーに「機能が壊れている」という印象を与えます。

**修正提案:**
- `calculateMatchRate`関数を修正し、`'main'`カテゴリに限定せず、**すべての材料（ingredients）**を対象にマッチ率を計算するように変更してください。調味料などを除外したい場合は、より堅牢な除外リスト（例：`unit`が`'g'`や`'ml'`の特定調味料リスト）を別途定義することを検討してください。

### 📱 iPhoneでの入力時における画面自動ズームの問題

**現状:**
`index.html`の`viewport`メタタグに`maximum-scale=1`が設定されていません。

**問題点:**
iPhoneのSafariでは、フォントサイズが16px未満の入力フィールドをタップすると、自動的に画面がズームされる仕様があります。これにより、ユーザーが文字入力をしようとするたびに意図しないズームが発生し、入力後もズームが戻らないため、非常にストレスの大きい操作を強いることになります。

**修正提案:**
- `index.html`の`<meta name="viewport" ...>`に`maximum-scale=1`を追加し、自動ズームを抑制してください。アクセシビリティの観点から非推奨とされることもありますが、今回のUIでは入力フィールドのフォントサイズを16px以上に保つことが難しいため、操作性向上を優先するべきです。
  ```html
  <meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1, viewport-fit=cover" />
  ```

### 🎨 在庫管理画面のUI/UXに関する複数課題

**現状:**
`src/components/StockManager.tsx`で実装されている在庫管理画面には、以下の問題があります。
1.  食材の数量を変更するには、まず右側のシェブロン（`v`）ボタンをタップして詳細を開き、それから数値を入力する必要がある。
2.  詳細を開いた状態では、数量入力欄と削除（ゴミ箱）ボタンが近接して配置されている。

**問題点:**
1.  数量を編集するという一般的な操作のために、2ステップ（開く→入力）が必要であり、直感的ではありません。特に多くの食材を一度に編集したい場合に手間がかかります。
2.  タップ領域が近いため、数量を編集しようとして誤って食材を削除してしまうリスクが高いです。

**修正提案:**
- **UIの抜本的見直し**を提案します。
  - 各食材項目を横スワイプすることで、削除ボタンや編集ボタンを表示するUIを検討してください。これは多くのモバイルアプリで採用されている直感的な操作です。
  - または、数量部分を直接タップ可能にし、モーダルダイアログやインラインでの数値入力ができるように変更します。トグルで表示を切り替える現在の方式は廃止すべきです。

### 📝 閲覧履歴機能の未実装

**現状:**
下部ナビゲーションに「履歴」タブが存在しますが、タップすると「履歴機能は準備中です」と表示され、機能が実装されていません。

**問題点:**
ユーザーが一度見たレシピを後から簡単に見返すことができず、アプリの利便性を大きく損なっています。ボタンが存在するのに機能しないのは、未完成であるという印象を強めます。

**修正提案:**
- レシピ詳細ページ(`RecipeDetail.tsx`)が表示された際に、そのレシピIDと閲覧日時を`IndexedDB`の新しいテーブル（例：`viewHistory`）に保存するロジックを追加します。
- 履歴ページでは、`viewHistory`テーブルから閲覧履歴を日付の降順で取得し、リスト表示します。同じレシピが複数回閲覧された場合は、最新の閲覧日時のみを表示するように重複を排除する処理も必要です。

### 👆 ヘッダーボタンのタップ領域の狭さ

**現状:**
`Header.tsx`内の設定ボタンやAIパースボタンなどのアイコンボタンは、`p-2`というクラスが指定されており、タップ可能な領域が比較的小さくなっています。

**問題点:**
特にモバイルデバイスでは、指で正確にタップするのが難しく、操作ミスを誘発する可能性があります。Fittsの法則に基づいても、ターゲットは大きい方が操作しやすくなります。

**修正提案:**
- 各ボタンのpaddingを`p-3`などに増やし、タップ領域を十分に確保してください。これにより、ユーザーはより快適に操作できるようになります。

### 🌿 ホーム画面への「旬の食材」機能の提案

**現状:**
現在のホーム画面は静的で、「ようこそ」というメッセージが表示されるだけです。

**問題点:**
アプリを起動した際のトップページにパーソナライズされた情報や発見がないため、ユーザーのエンゲージメントを高める機会を逃しています。

**修正提案:**
- 現在の月や季節に応じた「旬の食材」リストを作成します。
- アプリ起動時に、その旬の食材を材料に含むレシピをいくつかピックアップし、ホーム画面に「今月のおすすめ」などとして表示する機能を実装してください。これにより、ユーザーは新しいレシピと出会うきっかけを得られます。

### 📦 データインポート戦略の抜本的見直し

**現状:**
`ImportPage.tsx`としてCSVファイルをユーザーが手動でインポートする機能が存在します。また、リポジトリ内には`AX-XA20_recipes_complete.csv`と`KN-HW24H_recipes_complete_complete.csv`という巨大なCSVファイルが同梱されています。

**問題点:**
- **ユーザーの手間**: ユーザーにCSVファイルを探させ、手動でインポートさせるのは非常に不親切です。
- **オフライン非対応**: アプリをインストールした初期状態ではレシピが一件も入っておらず、オフラインで利用を開始できません。

**修正提案:**
1.  **CSVインポート機能の完全な廃止**: `ImportPage.tsx`および関連するUI（ヘッダーのアップロードボタンなど）を削除します。
2.  **レシピデータの事前ビルド**: ビルドプロセス中に、リポジトリ内の2つのCSVファイルを読み込み、JSON形式に変換してアプリにバンドルします。
3.  **初回起動時のDB投入**: アプリの初回起動時に、バンドルされたJSONデータを`IndexedDB`に自動的に投入する処理を`initDb.ts`などに追加します。これにより、ユーザーはインストール後すぐに全レシピをオフラインで閲覧・検索できるようになります。
