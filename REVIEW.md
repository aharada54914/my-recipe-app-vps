# レシピアプリ プロジェクトレビュー

## 1. はじめに

本レビューは、QAエンジニアの視点から、プロジェクトの品質、パフォーマンス、そしてユーザー体験を評価し、改善点を指摘するものです。

全体として、仮想スクロールや検索の最適化など、パフォーマンスに関する多くのベストプラクティスが実装されており、技術的な基盤は堅固であると評価します。しかし、PWAとしての根幹機能に重大な欠陥が見られ、その他にも修正すべき点が散見されます。以下に具体的な指摘事項を詳述します。

## 2. 最重要指摘事項 (Blocker)

### 🚨 PWAの根幹機能（Service Worker）の完全な欠落

**現状:**
プロジェクト全体を調査した結果、Service Workerを登録・管理するためのコードが一切含まれていません。`vite.config.ts`に`vite-plugin-pwa`のようなPWAプラグインは導入されておらず、`main.tsx`や`index.html`にもService Workerを登録する処理が見当たりません。

**問題点:**
これは**PWA (Progressive Web App) として致命的な欠陥**です。Service Workerが存在しないため、以下のPWAの核となる機能が一切動作しません。

- **オフライン動作**: ネットワーク接続がない環境でアプリが全く起動・利用できません。
- **キャッシュによる高速化**: アセット（JS, CSS, 画像など）がキャッシュされず、毎回ネットワークからダウンロードされるため、起動時間や動作速度が著しく低下します。
- **プッシュ通知**: 将来的な機能拡張の妨げとなります。

`manifest.json`が設定されているだけではPWAとは呼べません。これは単なる「ホーム画面に追加できるウェブサイト」に過ぎず、ユーザーが期待するアプリのような体験（信頼性、速度）を提供できません。

**修正提案:**
1.  `vite-plugin-pwa`をプロジェクトに導入します。
    ```bash
    npm install vite-plugin-pwa -D
    ```
2.  `vite.config.ts`を更新し、PWAプラグインを設定します。キャッシュ戦略（`strategies`）を定義し、`manifest`をインライン化することが推奨されます。
3.  `index.html`から既存の`<link rel="manifest" ...>`を削除し、プラグインによる自動注入に任せます。
4.  Service Workerの自動更新を促すUIコンポーネントを実装します。

## 3. 重要指摘事項 (Critical)

### 🎨 アプリアイコンがデフォルトのまま

**現状:**
`public/manifest.json`および`index.html`で指定されているアイコンが、Viteのデフォルトである`/vite.svg`のままです。

**問題点:**
アプリアイコンは、ユーザーがホーム画面でアプリを識別するための最も重要な要素です。デフォルトのままでは、アプリの品質と信頼性を著しく損ないます。ユーザーに未完成品であるという印象を与えかねません。

**修正提案:**
- アプリのブランドを象徴する、適切なサイズのアイコンセット（192x192, 512x512など）を`public/icons`ディレクトリなどに配置します。
- `vite-plugin-pwa`の設定内で、これらのアイコンを`manifest`の`icons`プロパティに正しく指定してください。

### 🔄 データベース初期化処理の冗長性

**現状:**
`App.tsx`内で定義されている複数のコンポーネント（`AppShell`, `RecipeDetailPage`, `AiParsePage`など）が、それぞれ`useEffect`内で`initDb()`を呼び出しています。

**問題点:**
データベースの初期化は、アプリケーションのライフサイクルで一度だけ実行すれば十分です。各ページコンポーネントで個別に呼び出すのは冗長であり、無駄な処理です。アプリケーションのエントリーポイントで一度だけ実行するべきです。

**修正提案:**
- `App.tsx`の最上位にある`App`コンポーネント、もしくは`main.tsx`で一度だけ`initDb()`を呼び出し、初期化が完了するまでローディング画面を表示する構造に変更してください。
- `Promise`をトップレベルで`await`するか、`useState`で初期化状態を管理し、子コンポーネントに渡すのが一般的なパターンです。

## 4. 軽微な指摘事項 (Minor)

### 🧮 塩分計算ロジックの丸め処理の不一致

**現状:**
`QA.md`では「10g単位の丸め」が要求されていますが、`src/utils/recipeUtils.ts`内の`calculateSalt`関数では、塩分量(`saltG`)が`Math.round(...) / 10`によって小数点第一位に丸められています。

**問題点:**
仕様書と実装の間にわずかな乖離があります。`formatQuantityVibe`関数で最終的な表示は10g単位に丸められるため、実質的な影響は少ないかもしれませんが、計算ロジックの正確性を担保するためには、仕様と実装を一致させるべきです。

**修正提案:**
- `calculateSalt`関数内の`saltG`の計算結果を、10g単位で丸めるのではなく、要求通り小数点第一位までの精度を保つようにするか、あるいはQAドキュメントの記述を「小数点第一位までの丸め」に修正するか、どちらかに統一してください。現状では`saltG`の計算結果が`12.3`のような値になる可能性がありますが、`formatQuantityVibe`を通すと`10g`になってしまい、情報が欠落します。塩分量そのものはより正確な値を保持し、表示の際に丸めるのが適切です。

### 🧹 `react-router-dom`の非効率な利用

**現状:**
`App.tsx`の`AppShell`コンポーネント内で、`BottomNav`のタブ変更時に`navigate()`関数を呼び出してURLを制御し、さらに`activeTab`という`useState`で表示するコンポーネントを切り替えています。

**問題点:**
これは`react-router-dom`の意図された使い方ではありません。URLに応じて表示するコンテンツを決定するのがルーティングライブラリの役割です。現在の実装は、状態（`activeTab`）とURLが二重にUIを制御しようとしており、複雑でバグの温床になりやすいです。

**修正提案:**
- `Outlet`とネストされたルートを使用して、レイアウトとページコンテンツを分離してください。
- `AppShell`をレイアウトコンポーネントとし、`Header`と`BottomNav`を含めます。ページ固有のコンテンツは`<Outlet />`で描画します。
- `BottomNav`では、`useLocation`フックで現在のパスを取得し、アクティブなタブを決定します。タブのクリックでは、状態を変更するのではなく、`<Link>`または`navigate`で対応するパスに遷移させるだけにします。

## 5. 総括

本プロジェクトは、多くの高度な最適化手法を取り入れており、そのポテンシャルは非常に高いと評価します。しかし、PWAとしての基本要件が満たされていない現状は看過できません。

まずは最重要指摘事項である**Service Workerの実装**を最優先で対応してください。これが完了して初めて、PWAとしてのスタートラインに立つことができます。

厳しい指摘となりましたが、これは本プロジェクトが持つ可能性を最大限に引き出すためのものです。これらの改善点を乗り越え、ユーザーに愛される素晴らしいアプリが完成することを期待しています。
