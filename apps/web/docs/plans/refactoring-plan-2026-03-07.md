# Refactoring Plan

最終更新: 2026-03-07

## 前提

- 最優先領域は 1 つに絞らず、安定性・週間献立 UX・検索/在庫 UX・見た目刷新をすべて実施する
- 主利用端末はスマホ
- Google 連携と Gemini は中核機能として扱う
- 既存ユーザーデータの完全互換は必須ではない
- 現行 Liquid Glass は維持しなくてよい

## 基本方針

順番は以下に固定する。

1. 壊れにくくする
2. 週間献立を主導線として作り直す
3. 検索/在庫を短時間入力前提で磨く
4. 見た目をスマホ最適化で再設計する

理由:

- 週間献立 UX と見た目を先に触ると、現状の責務集中コードでは回帰が増える
- Google / Gemini が中核なので、失敗時 UX と非同期処理の整理を先にやる必要がある
- スマホ専用前提なら、情報量より操作速度と状態の分かりやすさを優先すべき

## 現状の主問題

### 1. 週間献立が巨大すぎる

- `src/pages/WeeklyMenuPage.tsx` に生成、入替、共有、QR、通知、天気、買い物、カレンダー登録が集中
- 画面責務とユースケース責務が分離されていない

### 2. データ層と副作用が密結合

- `src/db/db.ts` に型、スキーマ、マイグレーション、互換処理が集中
- `src/contexts/PreferencesContext.tsx` と `src/hooks/useGoogleDriveSync.ts` に起動時副作用が混在

### 3. UI フィードバックが古い

- `alert` / `confirm` に依存
- 成功、失敗、要再試行、同期中の違いが画面上で連続的に表現されない

### 4. スマホ導線が整理不足

- ホームの情報密度が高く、優先導線が曖昧
- BottomNav がアイコン中心で意味が弱い
- 固定アクションバーと本文の役割が重複

## 実装フェーズ

## Phase 0: 安定化基盤

目的:

- 以降の UI 改修で壊さない状態を作る

作業:

- brittle test を修正する
- 週間献立、検索、在庫、Google 連携、Gemini の E2E を画面単位に分割する
- `alert` / `confirm` / `prompt` 使用箇所を一覧化し、Toast / Inline message / Modal confirm に移す方針を決める
- エラー種別を `validation`, `network`, `auth`, `unknown` に整理する

完了条件:

- `npm test` と `npm run test:smoke:ci` が安定
- 主要画面で空状態、通信失敗、権限不足の UI ケースがある

## Phase 1: アーキテクチャ整理

目的:

- 画面からロジックを剥がし、変更しやすくする

作業:

- `src/db/db.ts` を以下へ分離
  - `src/domain/`
  - `src/db/schema/`
  - `src/db/migrations/`
  - `src/repositories/`
- Dexie 直呼びを画面から減らし、repository / service 経由に寄せる
- `PreferencesProvider` の起動副作用を app startup service へ移す
- Drive 同期を `backup coordinator` と `restore coordinator` に分離する
- 起動シーケンスを `initDb -> migration -> seed -> background maintenance` として明文化する

完了条件:

- UI コンポーネントが DB 詳細を知らない
- 起動副作用の失敗が UI 上で追跡可能

## Phase 2: 週間献立 UX 再設計

目的:

- このアプリの主機能を、スマホで迷わず使える状態にする

作業:

- `WeeklyMenuPage` を分割
  - `useWeeklyMenuController`
  - `useWeeklyMenuGeneration`
  - `useWeeklyMenuWeather`
  - `useWeeklyMenuSharing`
  - `useWeeklyMenuCalendar`
- 画面を 3 セクションに再編
  - 週ヘッダー
  - 今週の状態サマリー
  - 日別カード
- 空状態と生成後状態で UI を切り替える
- 固定アクションバーを再設計
  - 未生成時は `献立を作る` を主 CTA 1 本に絞る
  - 生成後は `再生成`, `買い物`, `共有`, `登録` に再編
- 推薦理由、栄養評価、天気は折りたたみではなく「概要 + 詳細導線」に変える
- QR / 共有 / カレンダー登録結果は blocking dialog ではなくステータスカード + toast へ変更

完了条件:

- 初見で「何をすればよいか」が 3 秒以内に分かる
- 1 週間分の確認と編集が片手操作で完結する

## Phase 3: 検索 / 在庫 UX 再設計

目的:

- スマホで最短操作にする

作業:

- 検索画面を「入力」「フィルタ」「結果」の 3 層に整理
- カテゴリ、機種、時短、旬のフィルタを chip ベースで再編
- 検索履歴と最近見たレシピを分離
- 在庫管理の独自スワイプ削除を廃止し、明示操作に変更
- 在庫入力を次の順で最適化
  - 候補から追加
  - +1 / -1
  - 数量直接編集
  - 最近使った食材
- 調味料プリセットを「初回セットアップ」と「日常追加」に分ける
- 検索結果の理由を短文化
  - 在庫一致
  - 好み補正
  - 旬
  - 時短

完了条件:

- 在庫 1 件追加が 3 タップ程度で終わる
- 検索結果の並び順に納得感がある

## Phase 4: Google / Gemini 中核体験の再設計

目的:

- 中核機能として信頼できる状態にする

作業:

- Google 連携を明示的な状態機械にする
  - 未設定
  - 未ログイン
  - 権限不足
  - 使用可能
  - 同期失敗
- Gemini も同様に整理
  - API キー未設定
  - 使用可能
  - 混雑 / 制限
  - 解析失敗
- 「使えない理由」と「次にやること」を常時 UI に出す
- URL import / 写真解析 / 献立再生成の失敗時に再試行導線を付ける
- AI 結果を保存前レビュー中心に統一する

完了条件:

- ログインや API キーの不足で、ユーザーが詰まらない
- 失敗したときに次アクションが画面に出る

## Phase 5: ビジュアル刷新

目的:

- Liquid Glass をやめても、スマホで見やすく早く操作できる見た目にする

推奨案:

- 方針名: Warm Tactile Kitchen
- 方向性:
  - 背景は半透明ガラスではなく、暖色寄りのマット面
  - カードはガラス感より、紙や調理ラベルに近い質感
  - アクセントはオレンジ単色依存をやめ、役割別に色を分ける
  - 情報は影より余白で区切る

トークン案:

- background: 生成り寄りの明色
- surface: 白ではなく少し温かい中間色
- accent-primary: 調理アクション用のオレンジ
- accent-fresh: 在庫 / 旬 / 成功用のグリーン
- accent-ai: Gemini 系の青緑
- warning: 黄土寄り
- error: 彩度を抑えた赤

UI 方針:

- BottomNav はラベル付きにする
- 主要 CTA は 1 画面 1 つに絞る
- セクション見出しは説明文を減らし、サマリー値を置く
- カード内の文字サイズ差を縮めて読みやすくする

代替案:

- A. Warm Tactile Kitchen
  - おすすめ
  - 料理アプリらしさが出しやすい
- B. Utility Mobile Sheet
  - 業務アプリ寄り
  - 可読性は高いが楽しさが弱い
- C. Editorial Recipe Journal
  - 情緒は出るが献立・在庫の操作画面とはやや相性が悪い

## Phase 6: 整理と削除

目的:

- 中途半端な互換処理を残さない

作業:

- 許容可能な範囲で古い互換コードを削除
- 使われていない localStorage キー、旧設定、重複コンポーネントを整理
- ドキュメントを現行実装に合わせて更新

## PR 分割案

1. テスト安定化と失敗ケース追加
2. DB / Repository / Startup 再編
3. 週間献立ロジック分離
4. 週間献立 UI 再設計
5. 検索 UX 改修
6. 在庫 UX 改修
7. Google / Gemini 状態設計
8. デザイントークン刷新
9. 全体 polish と不要コード削除

## 成功指標

- 週間献立作成完了までの平均タップ数減少
- 在庫 3 品登録までの所要時間短縮
- 共有 / QR / カレンダー登録の失敗再試行率改善
- E2E の安定化
- 初回ホーム表示での主要導線クリック率改善

1. brittle test 修正
2. 週間献立の controller 分離
3. `alert` / `confirm` の UI 置換基盤作成
4. 週間献立の空状態と固定アクションバーを作り直す
