# 統合マスターリファクタリング計画

## 1. はじめに

このドキュメントは、以下の2つの計画を統合し、技術的な依存関係を考慮した最適な実装順序を提示するものです。

- **統合リファクタリング計画v4**: ユーザー体験を向上させる機能改善
- **エキスパートレベル・リファクタリング計画**: プロジェクトの信頼性、保守性、開発体験を向上させる技術的負債の返済

## 2. 依存関係の分析と実装戦略

**基本戦略**: まず、プロジェクトの基盤を強化する「エキスパートレベル計画」を優先的に実施します。これにより、後の機能追加がより安全かつ効率的に行えるようになります。

**依存関係**: 
- **グローバル状態管理 (Zustand)** は、多くの機能（在庫管理、ユーザー設定、献立）で利用されるため、最優先で実装する必要があります。
- **テストカバレッジの向上**と**CI/CD**は、すべての機能開発と並行して進めるべきです。

## 3. 統合実装計画（全12ステップ）

| フェーズ | ステップ | タスク | 目的 | 時間 |
| :--- | :--- | :--- | :--- | :--- |
| **A: 基盤強化** | 1 | **グローバル状態管理の導入 (Zustand)** | 状態管理を中央集権化し、後の開発を効率化 | 3時間 |
| | 2 | **ルーティングの完全移行** | URL駆動の設計を完成させ、コンポーネントの責務を明確化 | 1.5時間 |
| | 3 | **テストカバレッジの向上 (Part 1)** | 既存の主要ロジックの品質を保証 | 2時間 |
| **B: 機能改善** | 4 | `selectWeeklyMenu`関数の改善 | 再生成のバリエーションを増やす | 1時間 |
| | 5 | 代替レシピモーダルの改善 | 検索機能を追加する | 2時間 |
| | 6 | `ShoppingListContent`の修正 | 買い物リストを正しく表示する | 1.5時間 |
| | 7 | `AskGeminiPage`の作成 | 在庫管理とAI機能を統合する | 3時間 |
| | 8 | ナビゲーションとルーティングの変更 | UIの変更をアプリ全体に反映する | 1時間 |
| | 9 | `MultiScheduleView`の検索改善 | 検索体験を統一する | 1時間 |
| | 10 | お気に入りメニュー機能の実装 | 献立作成を効率化する | 2.5時間 |
| **C: 開発体験向上** | 11 | **Storybookの導入** | UIコンポーネントをカタログ化し、再利用性を高める | 2.5時間 |

**合計所要時間: 約23時間**

## 4. 詳細な実装手順

### フェーズA: 基盤強化 (6.5時間)

1.  **ステップ1: グローバル状態管理の導入 (Zustand)** (3時間)
    -   `pnpm install zustand`
    -   `src/stores`ディレクトリを作成し、`stockStore.ts`, `settingsStore.ts`, `weeklyMenuStore.ts`などを定義します。
    -   `StockManager`, `SettingsPage`, `WeeklyMenuPage`などで、`useState`や`useLiveQuery`をZustandのフックに置き換えます。

2.  **ステップ2: ルーティングの完全移行** (1.5時間)
    -   `MainLayout.tsx`を作成し、`Header`と`BottomNav`を含むレイアウトを定義します。
    -   `App.tsx`のルーティングを、`MainLayout`を使う形に修正します。

3.  **ステップ3: テストカバレッジの向上 (Part 1)** (2時間)
    -   `weeklyMenuSelector.ts`, `geminiParser.ts`など、既存の主要な`utils`ファイルのユニットテストを作成します。

### フェーズB: 機能改善 (11時間)

4.  **ステップ4: `selectWeeklyMenu`関数の改善** (1時間)
    -   `weeklyMenuSelector.ts`で、レシピを全件取得するように修正します。

5.  **ステップ5: 代替レシピモーダルの改善** (2時間)
    -   `WeeklyMenuPage.tsx`に検索バーを追加し、`Fuse.js`で代替レシピを検索できるようにします。

6.  **ステップ6: `ShoppingListContent`の修正** (1.5時間)
    -   `EditableShoppingList.tsx`コンポーネントを作成し、買い物リストの表示・編集機能を実装します。

7.  **ステップ7: `AskGeminiPage`の作成** (3時間)
    -   `AskGeminiPage.tsx`を新規作成し、URLからのレシピインポート、在庫からの献立提案、在庫管理の3つの機能を統合します。

8.  **ステップ8: ナビゲーションとルーティングの変更** (1時間)
    -   `BottomNav.tsx`の「在庫」を「Ask Gemini」に変更し、`App.tsx`のルーティングを更新します。
    -   `Header.tsx`からGeminiボタンを削除します。

9.  **ステップ9: `MultiScheduleView`の検索改善** (1時間)
    -   `MultiScheduleView.tsx`の検索ロジックを`Fuse.js`に統一します。

10. **ステップ10: お気に入りメニュー機能の実装** (2.5時間)
    -   `WeeklyMenuPage.tsx`に、お気に入りレシピを横スクロール表示し、曜日に追加できるUIとロジックを実装します。

### フェーズC: 開発体験向上 (4.5時間)

11. **ステップ11: Storybookの導入** (2.5時間)
    -   `npx storybook@latest init`を実行します。
    -   `RecipeCard`, `SearchBar`など、主要なUIコンポーネントのストーリーを作成します。

12. **ステップ12: CI/CDパイプラインの構築** (2時間)
    -   `.github/workflows/ci.yml`を作成し、`pnpm test`と`pnpm run lint`を自動実行するように設定します。
    -   （オプション）`deploy.yml`を作成し、Vercelへの自動デプロイを設定します。

## 5. まとめ

この統合マスタープランは、**機能追加**と**技術的負債の返済**を両立させるための最適なロードマップです。フェーズAで基盤を固めることで、フェーズBの機能改善がスムーズかつ安全に進められます。最後にフェーズCで開発体験を向上させることで、今後の継続的な開発が加速します。

この計画を完了することで、`my-recipe-app`は、ユーザー体験と技術的卓越性の両方を兼ね備えた、真のエキスパートレベルのアプリケーションとなります。
